# 评测与对抗性验收

## 评测单位

每个用例包含输入工作簿、配置版本、任务提示、预期输出和可验证断言。评测必须在干净上下文中运行，不向执行者泄露预期答案。

## 正常用例

- 续费前后人数变化正确。
- 往期达成率正确作用于新增学员。
- 搭班教师需求、共享教室类型和三种情景正确。
- 最小班容、目标班容和最大班容共同约束班级数，边界人数能给出正确开班状态。
- 招聘与扩租建议的提前月份、持续期和证据键正确。
- 相同输入、配置和数据日期重复运行得到相同 JSON 与 Markdown。

## 对抗用例

- 缺少必填工作表、表头被改名或关键单元格为空。
- 学员标识重复，续费率大于 1 或小于 0。
- 班容顺序错误、负容量、月份格式错误。
- 同一教师或教室在同一月份和时段重复占用。
- 输入含公式、外部链接或以 `= + - @` 开头的文本。
- 数据日期超过允许时效。
- 历史目标为零或缺失，导致招生达成率不可计算。

## 判分

- 机械断言由测试脚本判定，必须给出实际值与期望值。
- 管理建议由业务负责人做盲审，重点看正确性、可执行性、证据充分性和是否越权。
- 关键安全断言失败即整次评测失败，不用平均分掩盖。

## 上线门槛

1. 自动测试全通过。
2. Skill 结构校验通过。
3. 敏感信息扫描通过。
4. 至少三个历史周期回测通过。
5. 一线管理者书面确认口径与输出模板。
